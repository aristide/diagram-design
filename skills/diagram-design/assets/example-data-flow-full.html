D4N Platform · Role-Scoped Data Flow · 2025

Source to Insight

How data moves through the D4N platform across four organisational roles — from raw ingestion to published analytics — with typed payloads and access boundaries at each handoff.

Pipeline Roles

  • ADM Data Admins create projects, assign LDAP group memberships, and set MinIO bucket policies before any data lands
  • ENG Data Engineers own ingest — pulling from web APIs, databases, and SFTP — and transform raw files into staging tables
  • SCI Data Scientists receive only anonymized, pre-approved data; they explore, model, and publish findings to Superset
  • CON Data Consumers query aggregated, published views via Trino with read-only access — no direct pipeline or model access

Data Types in Flight

  • WB Web / Public — source APIs, government open-data portals, scraped public datasets entering the collect step
  • DB Dataset / Raw — raw file landing zones in MinIO; schema-on-read, no cleaning applied
  • TB Table / Analysis-ready — Trino-queryable Iceberg tables in staging and gold zones; typed, partitioned, documented
  • FL File / Report — model outputs, serialised notebooks, and dashboard exports published by Scientists

Access Boundaries

  • Anonymisation gate sits between Engineer clean-and-stage and the Scientist lane — no PII crosses that edge
  • Scientists never touch raw or staging zones; access is scoped to the anonymised gold bucket only
  • Consumers are bound to published views — no model internals, no intermediate tables, no raw paths
  • LDAP group policies enforced at MinIO bucket level on every zone; Trino row-level filters enforce the same boundary at query time
